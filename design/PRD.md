# Simple AI Writer - 产品需求文档 (PRD)

---

## 1. 概述与产品定位

### 1.1 产品背景
许多小说创作者、剧本作家和技术写作者在创作过程中，需要维护一份庞大且复杂的“设定集”（人物性格、世界观背景、魔法规则、道具功能等）。市面上现有的 Markdown 编辑器（如 Obsidian, Typora）虽然排版精美，但缺乏智能化的设定校验与写作辅助；而现有的 AI 写作工具往往是通用聊天接口，无法深度理解创作者自定义的庞大世界观，导致 AI 生成的内容经常“出戏”或产生设定冲突。

### 1.2 产品定位
**Simple AI Writer** 是一款专为创作者打造的、多平台的、极致简洁的 **Local-first（本地优先）Markdown 写作软件**。它深度集成了**设定知识库（Writing Resource Library）**与 **AI 写作助手**，利用检索增强生成（RAG）技术，让 AI 能够“阅读”并“严格遵循”创作者的自定义设定来辅助写作、校验逻辑、润色文本。

### 1.3 核心设计原则
*   **极致简洁 (Simplicity First)**: 界面无干扰，专注于写作本身。
*   **本地优先 (Local-first & Privacy)**: 用户的文稿和设定数据完全保存在本地，保护知识产权。
*   **设定契合 (Lore-friendly AI)**: AI 的任何续写和校验均与用户的设定库深度联动。
*   **多平台一致性 (Cross-platform)**: 兼容 Windows, macOS, Linux, iPad，且拥有高度一致的流畅体验。

## 2. 目标平台与技术架构

### 2.1 目标平台与适配策略
*   **桌面端**: Windows (10/11), macOS (11+), Linux (Ubuntu 20.04+)。
*   **移动/平板端**: iPadOS (15+)，深度适配 Apple Pencil 触发与手势触控。
*   **技术选型建议**:
    *   **应用框架**: **Tauri v2**。相比 Electron，Tauri 具有极小的安装包体积（<20MB）、超低的内存占用，且 v2 版本原生支持 iOS/iPadOS 和 Android 编译。
    *   **前端框架**: React + Vite + TypeScript。
    *   **样式系统**: Vanilla CSS (CSS Variables) + CSS Modules，以实现极致精细的视觉微调和磨砂玻璃（Glassmorphism）效果。

### 2.2 数据存储与工程目录架构
本系统采用类似于 Visual Studio Code 的基于文件夹的工程管理，所有资料库、文档、图片等都存储在工程文件夹下的指定目录中。

```
项目工程根目录/
├── 📁 writing/          <-- 写作目录：存放用户创作的 Markdown 章节文件
│   ├── 第一章.md
│   └── 第二章.md
├── 📁 output/           <-- 输出目录：存放导出的 PDF, HTML 等文件
└── 📁 .ai-writer/        <-- 配置与资源库目录：存放元数据与设定库
    ├── project.db       <-- SQLite 数据库 (标签, 检索索引, 配置等)
    └── 📁 lore/         <-- 写作资源库目录
        ├── 📁 characters/     <-- 人物/生物/机械设定目录
        │   └── 📁 elden/      <-- 特定人物文件夹 (以目录方式组织)
        │       ├── index.md   <-- 主描述文件 (基本卡片属性、主干摘要)
        │       ├── backstory.md  <-- 补充文件：身世背景描述
        │       ├── relationship.md <-- 补充文件：社会人际关系
        │       ├── avatar.png <-- 本地配图/头像
        │       └── sketch.jpg <-- AI 辅助生成的设定插图
        ├── 📁 world/          <-- 世界观设定目录 (目录组织)
        │   └── 📁 middle_earth/
        │       ├── index.md
        │       └── history.md
        ├── 📁 factions/       <-- 组织与势力设定目录
        ├── 📁 items/          <-- 物品道具设定目录
        └── 📁 skills/         <-- 技能魔法设定目录
```

### 2.3 AI 模型支持与配置

#### 2.3.1 模型供应与管理层级
本产品使用 **Bring Your Own Key (BYOK)** 模式。模型管理遵循以下层级：
*   **Provider (供应商)**: 定义供应商、API URL、API Key 以及对应的 API 标准。
*   **Model Management (模型管理)**: 每个 Provider 可提供多个模型，用户可在配置页对模型进行细化管理：
    *   **字段定义**: 模型 ID、模型名称、模型类型（文本、多模态、图片、视频）。模型类型会直接影响该模型可用的功能范围（例如插图生成功能只能选择图片模型）。
    *   **计费信息配置**: 支持用户输入各模型的计费单价，包含输入 token 价格、输入命中缓存价格、输出 token 价格。以每 1M (1,000,000) Tokens 为单位进行配置与统计，方便软件在底部状态栏和 AI 面板实时统计用户的写作成本。

#### 2.3.2 预置模型标准配置
系统预置主流模型规范，支持以下标准供应商及特定模型：
*   **OpenAI API 标准**:
    *   **标准文本/多模态模型**: **GPT5** (作为生成、润色及一致性逻辑校验的标准模型)。
    *   **生图/插图模型**: **Image2** (作为插图生成的标准模型)。
*   **Google Gemini API 标准**:
    *   **高级推理/文本模型**: **Gemini 3.1 pro** (用于超长上下文推理、设定校验以及大纲生成)。
    *   **快速响应/多模态模型**: **Gemini 3.5 flash** (用于日常高频的续写、改写、错别字修正及辅助编辑)。
    *   **生图/插图模型**: **Nano banana** (作为轻量或专属的图像生成模型)。
*   **兼容 OpenAI API 的第三方模型**:
    *   例如本地部署的 **Ollama** 离线模型，用于保护 100% 隐私的本地创作场景。

## 3. 功能需求详细说明

### 3.1 基础写作与文档管理

#### 3.1.1 现代化 Markdown 编辑器
*   **语法规范**: 支持 GitHub Flavored Markdown (GFM) 完整规范。
*   **数学公式与图表**: 支持 KaTeX 公式渲染、Mermaid 流程图渲染。
*   **实时预览 (Live Preview)**:
    *   默认采用**所见即所得 (WYSIWYG)** 模式（类似 Typora）：当光标离开 Markdown 标记时，自动渲染为富文本效果；光标移入时展示原始标记。
    *   支持**分栏预览模式**（左侧编辑，右侧渲染）和**纯文本源码模式**。
*   **写作专注模式**:
    *   **打字机滚动 (Typewriter Scrolling)**: 输入行始终保持在屏幕中央。
    *   **段落聚焦 (Focus Mode)**: 当前输入段落正常显示，非活动段落透明度降低至 30%。
    *   **无边框全屏模式**: 隐藏所有侧边栏与系统状态栏。

#### 3.1.2 文档与章节目录管理
*   **工程脚手架与初始化 (Folder Opening & Scaffolding)**:
    *   采用类似于 VS Code 的“打开文件夹”方式管理项目。如果用户打开的文件夹是一个空目录，系统会自动初始化并生成 `writing/`、`output/` 以及隐藏的 `.ai-writer/` 目录结构。
*   **项目文件树**: 支持多层级文件夹结构，管理 `writing/` 文件夹下的 Markdown 文稿。
*   **章节大纲导航**: 自动解析当前文档中的 Markdown 标题 (`#` 至 `######`)，在右侧侧边栏生成实时更新的树形大纲，点击标题可平滑滚动至对应位置。
*   **标签与分类系统**:
    *   支持多标签检索，支持跨文档批量添加标签。
    *   支持在文档 Frontmatter 中定义 `tags: [标签1, 标签2]`，编辑器自动解析。
*   **文稿快照与备份历史 (Draft Snapshots & Recovery)**:
    *   系统会自动监控 `writing/` 文件夹下 Markdown 文稿的变化，并在用户静止输入 5 秒后或关闭文件时，在 `.ai-writer/project.db` 中自动创建增量快照。
    *   提供本地“回滚历史记录”面板，保存最近 30 天的文稿修改历史，确保即使 Markdown 文件在外部被破坏，创作者的数据也绝对安全。

#### 3.1.3 导出功能
*   支持一键导出当前文档或整本书籍至 `output/` 文件夹下：
    *   **PDF**: 提供“学术”、“文学小说”、“极简现代”等多种排版模板。
    *   **HTML**: 生成包含完整样式表（CSS）的自包含单文件。
    *   **Markdown**: 导出纯净的、剥离了本地资源链接的 Markdown 打包文件。

### 3.2 写作资源库 (Lore Knowledge Base)

写作资源库以**目录结构**对设定进行存储和管理。每个设定条目都有其专属文件夹，允许存放多个描述文件和相关多媒体资源，而非单薄的纯文本。

#### 3.2.1 设定分类与目录结构
资源库预设以下 6 大标准分类目录：
1.  **characters/** (人物/生物/机械设定): 如 `lore/characters/elden/`。
2.  **world/** (世界观设定): 如 `lore/world/magic_rules/`。
3.  **factions/** (组织与势力): 如 `lore/factions/templars/`。
4.  **items/** (物品与道具): 如 `lore/items/sky_sword/`。
5.  **skills/** (技能与魔法系统): 如 `lore/skills/fireball/`。
6.  **custom/** (其他设定): 支持用户自定义创建分类（如 `lore/custom/chronology/`）。

#### 3.2.2 实体文件夹内容规范
每个实体的专属文件夹内，至少包含以下文件：
*   **`index.md` (核心描述文件)**: 必须存在。定义实体的核心 Frontmatter 信息（名称、同义词别名、分类、简短摘要），并使用 Markdown 描述其主干设定。
    ```markdown
    ---
    name: 艾尔登
    aliases: [黄金之子, 艾尔登领主, Elden]
    category: characters
    summary: 极北之地的年轻法师，拥有操控冰霜的血脉。
    ---
    # 艾尔登
    艾尔登是本作的主角，出生于...
    ```
*   **补充描述 Markdown 文件 (`*.md`)**: 用户可以按需创建多个补充文件（例如 `backstory.md` 记录身世背景，`relationship.md` 记录人际纽带），使单一设定可以极其庞大和具有条理。
*   **静态媒体资源 (图片/音效等)**:
    *   `avatar.png` / `avatar.jpg`: 角色头像或实体主图。
    *   `concept_art.png`: 场景设定或 AI 辅助生成的设定插图。
*   **静态资源引用与解析规范 (Asset Reference & Preview Resolution)**:
    *   **外部编辑器兼容性**: 在 Markdown 中引用设定库中的图片时，系统必须自动使用基于当前文稿的物理相对路径（例如：`![艾尔登头像](../.ai-writer/lore/characters/elden/avatar.png)`）。这可确保用户使用外部编辑器（如 Obsidian/Typora）打开该项目文件夹时，图片仍然能正常显示。
    *   **本地安全渲染协议 (Custom Tauri Protocol)**: 为了遵守 Tauri 和浏览器的内容安全策略 (CSP)，前端编辑器渲染本地图片时，不使用 `file://` 协议，而是通过 Tauri 后端注册一个自定义安全协议（例如 `ai-writer-asset://`），自动将请求隐式映射到物理工程目录中的对应文件。
*   **`meta.json` (可选的系统元数据)**: 记录创建时间、修改时间以及用于 RAG 的本地向量化缓存状态。

#### 3.2.3 变更历史记录 (Version History)
*   系统会监控实体文件夹的变更。每当用户编辑或 AI 更新了实体目录下的 `*.md` 文件时，系统自动在本地生成一个轻量级的 JSON 差异（Diff）记录。
*   用户可以在实体详情面板中查看该设定的历史版本记录，并支持一键回滚到特定时间点。

### 3.3 AI 写作助手 (AI Assistant)

AI 写作助手通过在向大模型（如 GPT5 或 Gemini 3.1 pro）发送请求时，动态检索并组装当前场景所需的设定内容，确保 AI 创作内容严格符合世界观和角色设定。

#### 3.3.1 核心写作任务
1.  **大纲扩写与段落生成**: 用户在编辑器中编写简要的场景发展，AI 自动读取涉及角色的性格、所处世界的规则，生成符合设定、行文流畅的章节内容。
2.  **智能设定一致性校验 (OOC 校验)**:
    *   检查已书写文稿中是否含有偏离设定库的情节（如已折断的长剑在后续章节再次被拔出，或已死亡的角色重新出场说话）。
    *   在编辑器右侧面板显示冲突警告卡片，高亮冲突文本并附带来源引证。
3.  **润色、改写、续写与摘要**:
    *   **续写**: 根据文稿结尾自动向后预测。
    *   **润色**: 切换写作风格（如“古风”、“暗黑”、“细腻”等预设 prompt）。
    *   **改写**: 针对选中段落进行叙事视角转换或句式精简。
4.  **AI 插图生成**:
    *   用户在文稿中选中描述段落，或在人物设定文件夹中，点击“生成插图”。
    *   系统调度 **Image2** 或 **Nano banana** 生成图片，自动下载并存储到对应设定文件夹下（如 `lore/characters/elden/concept_art.png`），并自动在 Markdown 中插入图片链接。

### 3.4 上下文记忆管理体系 (Context Memory Management System)

为了解决长篇创作时 AI 容易遗忘前文的问题，同时规避在超长上下文中消耗高昂的 Token 费用（特别是像 GPT5 这种高阶模型），系统内置了一套**完善的上下文记忆管理体系**。

```
┌────────────────────────────────────────────────────────┐
│                    大语言模型 (LLM)                     │
└──────────────────────────▲─────────────────────────────┘
                           │ 动态组装 Context
┌────────────────────────────────────────────────────────┐
│                   上下文记忆管理器 (CMM)                 │
├────────────────────────────────────────────────────────┤
│ 1. 固定的全局设定 (Synopses, System Rules)              │
├────────────────────────────────────────────────────────┤
│ 2. 命中缓存的设定库信息 (Gemini Context Cache 优化)      │
│    ├─ 🔍 精准实体匹配: [艾尔登/index.md] (仅提取index.md)│
│    └─ 🌐 相似度召回: 动态提取 backstory.md              │
├────────────────────────────────────────────────────────┤
│ 3. 动态文稿上下文                                       │
│    ├─ 📝 极近文稿: 滑动窗口 (最近 3,000 字原文)         │
│    └─ 🗒️ 历史章节: 递进式 AI 场景摘要 (每章 200 字)       │
└────────────────────────────────────────────────────────┘
```

#### 3.4.1 上下文分层组装策略
在用户触发 AI 请求时，记忆管理器将上下文划分为四个层级，并按优先级动态拼接：

1.  **全局核心配置与大纲 (Global Core, 优先级 1)**:
    *   系统 Prompt（设定 AI 的写作角色、行文语气、约束指令）。
    *   整部作品的总大纲与故事主线描述（建议控制在 500-1000 tokens）。
2.  **局部设定检索 (Lore Retrieval - RAG, 优先级 2)**:
    *   **别名精准匹配**: 扫描当前编辑区文稿，匹配实体别名（如文中出现了“黄金之子”，则触发“艾尔登”的检索）。
    *   **向量相似度召回**: 使用本地 SQLite 向量检索。向量化与检索的颗粒度**仅到达写作资源库的词条（Entity/设定目录）级别**，而不是对文本进行细碎的切片（Chunking）。系统将每个实体的名称、别名及 `index.md` 整体向量化为单个向量，检索时一旦命中，即召回该实体目录下的对应描述，从而确保召回上下文的结构完整性与产品设计的极致简洁。
        *   **离线向量化计算方案 (Offline Embedding Fallback)**:
            *   若用户选择在线模型，系统默认通过 OpenAI/Gemini Embeddings API 计算向量。
            *   若用户选择本地 Ollama 离线写作，系统将支持通过 Ollama 的本地 Embeddings 接口（如配置 `nomic-embed-text` 模型）进行向量化。
            *   系统亦可内置一个纯本地的、轻量级 CPU 嵌入模型（例如利用 WebAssembly 或 ONNX 运行一个极小的 Bert-Embedding-v2），实现即使无网络且未配置 Ollama 时也能 100% 本地完成向量计算。
    *   **按需精简读取**:
        *   默认仅读取实体文件夹下的 `index.md`（核心摘要）。
        *   只有当匹配度极高或用户明确要求“深度参考”时，才加载补充描述文件（如 `backstory.md`）。这可以大幅节约 Token 消耗。
3.  **滑动窗口文稿原文 (Sliding Window Draft, 优先级 3)**:
    *   仅将光标位置前最近的 $N$ 个 Token（例如最近的 3,000 字原文）作为“即时叙事上下文”发送，以保持行文在语气和物理细节上的连贯。
4.  **历史章节摘要链 (Incremental Summary Chain, 优先级 4)**:
    *   在当前章节之前的历史章节，不发送原始全文，而是发送**递进式场景摘要**。
    *   系统会自动为完成的每一章生成一个 150 - 250 字的 AI 简短摘要，并拼接成“剧情进展链条”作为上下文。这既保证了 AI 对长线剧情的记忆，又压缩了 90% 以上的历史文稿 Token。

#### 3.4.2 缓存与性能优化 (Context Caching & Cache Stabilization)
*   **Gemini Context Caching 深度适配**:
    *   对于支持 Context Caching 的模型（如 **Gemini 3.1 pro** / **Gemini 3.5 flash**），系统采用“前缀缓存”机制。系统会将“全局核心配置（System Prompt）”、“整个设定库的 `index.md` 集合”以及“历史章节摘要链”作为**缓存前缀 (Cached Prefix)** 写入 API 缓存。
    *   因为设定库和历史摘要变化频率极低，后续的高频交互（如续写、润色、局部校验）仅需传入最新的滑动窗口文稿（作为 Dynamic Suffix），能极大提升响应速度，并享受命中缓存时的 Token 价格优惠。
*   **缓存抖动与命中率优化 (Cache Stabilization Design)**:
    为了防止因检索卡片频繁轮换或排序变化导致前缀失效（Cache Miss/抖动），引入以下三项设计：
    1.  **动静分离架构 (Separation of Static and Dynamic Context)**: 缓存前缀中仅包含静态及低频修改的信息（系统提示、总大纲、全部实体的 `index.md` 摘要）。高频变动的补充文件（如具体的 `backstory.md`）和当前编辑文稿不进入缓存前缀，作为动态后缀发送。
    2.  **确定性排序 (Deterministic Ordering)**: 所有被送入缓存前缀的设定实体（如 `index.md` 列表）或章节摘要，在拼接时必须按照**实体 ID/名称的字典序（Alphabetical Order）**进行确定性排序，绝不能按照向量相似度评分（Score）排序。这可以确保两次检索即使评分微调，拼接出的前缀文本也完全一致，从而 100% 命中缓存。
    3.  **懒轮换与缓冲池机制 (Lazy Rotation & Hysteresis Buffer)**: 对于必须包含在上下文中的动态细节设定，系统不进行“即时换入换出”。而是引入一个“活跃设定缓冲池”（容量为 3~5 个实体）。只有当某实体的向量相似度或别名匹配持续 $K$ 次编辑（例如用户输入 500 字以上，或进行了 5 次以上的 AI 交互）都未被触发，且有新实体需要入池时，才按照 LRU 算法将其移出。这有效防止了因光标在段落间移动导致缓存频繁被破坏（Cache Thrashing）。
*   **语义去重与局部重复检测**:
    *   当进行“一致性校验”时，如果检测到某段文字与上一次校验时相比未发生实质性更改，则直接返回本地 SQLite 缓存的校验结果，不再向 LLM 发起重复请求。

#### 3.4.3 Token 预算与用户可视面板
*   **Token 限制策略**: 用户可以在设置中配置“AI 记忆深度”级别：
    *   *极简 (Token-Saving)*: 滑动窗口为 1500 字，设定库仅召回 Top 2 核心卡片 `index.md`，不附带历史摘要。
    *   *标准 (Balanced)*: 滑动窗口为 3000 字，设定库召回 Top 5 相关卡片，包含历史摘要。
    *   *深邃 (Deep-Lore)*: 滑动窗口 8000 字，召回相关实体的全部 md 描述文件，包含完整大纲和摘要链。
*   **实时 Token 指示器**: 在编辑器底部状态栏和 AI 交互面板，提供**Token 预算仪表盘**。在用户发送请求前，实时估算当前上下文大小、预计消耗 Token 数以及当前模型单价折算出的预估费用，让创作者对成本了然于心。

## 4. UI/UX 界面设计规范

### 4.1 视觉系统 (Visual Tokens)
*   **色调体系**:
    *   **暗黑模式 (首选推荐)**: 背景采用深邃的太空黑（`#0D0F12`），前景色为柔和的暖白（`#E2E8F0`），强调色使用带有呼吸感的光晕蓝（`#3B82F6` 到 `#8B5CF6` 的渐变）。
    *   **日间模式**: 背景采用纸质暖白（`#FAF9F6`），文字为炭黑（`#1F2937`）。
*   **材质特征**:
    *   侧边栏、浮动对话框、设置面板采用微透明毛玻璃效果（Glassmorphism），结合磨砂模糊，增加视觉纵深。
*   **多语言与主题系统 (Theme & i18n)**:
    *   UI 框架深度适配多语言国际化，首期完整支持**简体中文**与**英文**的实时切换。
    *   支持“亮色”、“暗色”及“随系统自动切换”三种主题模式。

### 4.2 布局结构
软件采用响应式伸缩布局，桌面端使用标准三栏布局，结合侧边 tab、底部状态栏来高效组织页面功能：
1.  **左侧栏 (可收起)**: 包含项目目录树（管理 `writing/` 文档及 `lore/` 设定目录）。
2.  **主编辑区**: 黄金阅读宽度 `800px`，支持双栏模式和所见即所得实时渲染。
3.  **右侧 AI 功能区**: 集成大纲树导航、AI 聊天面板、一致性冲突诊断列表和当前命中设定实体的卡片浮窗。

## 5. 非功能性需求

## 6. 验证与验收标准

### 6.1 单元与集成测试策略
*   **Lore 目录扫描测试**: 确保新建实体目录及增加补充文件（如 `elden/backstory.md`）后，系统能自动扫描并重构向量检索索引。
*   **上下文裁剪测试**: 编写模拟数据（10万字草稿，50个设定文件夹），触发续写，检验发送给 LLM 的 Token 总量是否被正确控制在用户设定的上限预算内。

### 6.2 用户验收场景 (UAT Case)
1.  **场景 1: 设定集目录化联动**
    *   用户在 `lore/characters/elden/` 目录下创建 `index.md`，写明“艾尔登擅长冰霜魔法”；新建 `relationship.md`，写明“艾尔登的导师是古尔”。
    *   在主编辑区输入：“艾尔登的导师走了过来，看着他...”。
    *   双击“导师”触发悬浮卡片，系统需自动联想并正确弹出“古尔”的相关设定卡片。
2.  **场景 2: 上下文控制与缓存命中**
    *   使用 Gemini 3.5 flash 模型续写。
    *   底层抓包监控，确认发送给 Google API 的请求包含了正确的 Context Cache 标识，且后续连续写作请求中的 cached tokens 命中率大于 80%。
